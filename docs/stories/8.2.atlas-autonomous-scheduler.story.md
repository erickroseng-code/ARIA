# Story 8.2: Atlas — Autonomous Scheduler Endpoint

## Status
Ready for Dev

## Story

**As a** operador da Aria usando GitHub Actions como scheduler,
**I want** um endpoint `POST /api/traffic/atlas/auto-optimize` que o GitHub Actions possa chamar periodicamente,
**so that** o Atlas analise campanhas autonomamente, tome decisões de otimização e as execute sem intervenção humana.

## Acceptance Criteria

1. **Endpoint:** `POST /api/traffic/atlas/auto-optimize` aceita body `{ workspaceId: string, dryRun?: boolean }` e retorna `{ actionsExecuted: ActionResult[], analysis: string, dryRun: boolean }`.
2. **Autenticação:** Endpoint protegido por header `Authorization: Bearer ${ATLAS_SCHEDULER_TOKEN}` (novo env var). Se token inválido → 401.
3. **Fluxo autônomo:** O endpoint carrega dados da conta (`getInsights` + `getCampaigns`), roda `atlasAutoAnalyze()` para obter análise LLM, extrai ações do texto da análise e as executa uma por uma com guardrails (da Story 8.1).
4. **Múltiplas ações:** Diferente do chat (uma ação por vez), o scheduler extrai e executa TODAS as ações recomendadas na análise (até máximo de 5 por ciclo).
5. **Resposta estruturada:** Cada `ActionResult` inclui `{ action, entityId, result, dryRun }` para auditoria futura (Story 8.3).
6. **Timeout seguro:** Se análise LLM demorar > 30s, retornar 408 com `{ error: 'Analysis timeout' }` sem executar ações.

## Scope

**IN:**
- Novo endpoint `POST /api/traffic/atlas/auto-optimize` em `traffic.routes.ts`
- Nova função `atlasSchedulerRun(workspaceId, dryRun, trafficService)` em `atlas-orchestrator.ts`
- Extração de múltiplas ações da resposta do LLM
- Autenticação via `ATLAS_SCHEDULER_TOKEN`

**OUT:**
- Não criar tabela de audit log (Story 8.3)
- Não enviar Telegram (Story 8.4)
- Não implementar Google Ads (futuro)
- Não alterar o `atlasChat()` existente (preservar chat interativo)

## Dependencies

- **Story 8.1** deve estar concluída (guardrails implementados antes de executar ações autônomas)

## Complexity Estimate

**T-Shirt:** Medium (M)
**Story Points:** 5
**Dev Time:** ~5 horas

## Business Value

Este é o core da autonomia: sem este endpoint, todas as otimizações são manuais. Com ele, o GitHub Actions pode chamar a cada hora e o Atlas otimiza campanhas 24/7.

## Risks

- **Risco:** LLM pode retornar ações inválidas ou mal-formatadas
- **Mitigação:** `extractAction()` já existe e é robusto; adicionar try/catch por ação individual
- **Risco:** Múltiplas ações em sequência podem conflitar (ex: pausar e depois atualizar budget da mesma campanha)
- **Mitigação:** Limite de 5 ações por ciclo + guardrails da Story 8.1

## Tasks / Subtasks

- [ ] **Task 1** — Criar `atlasSchedulerRun()` em `atlas-orchestrator.ts` (AC: 3, 4, 5)
  ```ts
  export interface ActionResult {
    action: string;
    entityId: string;
    result: string;
    dryRun: boolean;
  }

  export async function atlasSchedulerRun(
    ctx: AtlasContext,
    trafficService: TrafficService,
    dryRun = true
  ): Promise<{ actionsExecuted: ActionResult[]; analysis: string }> {
    // 1. Pegar análise do LLM
    const analysis = await atlasAutoAnalyze(ctx, trafficService);

    // 2. Extrair todas as ações do texto (múltiplas)
    const actions = extractAllActions(analysis); // nova função

    // 3. Executar até 5 ações com guardrails
    const actionsExecuted: ActionResult[] = [];
    for (const action of actions.slice(0, 5)) {
      if (action.action === 'none') continue;
      const result = await executeAction(action, ctx, trafficService, dryRun);
      actionsExecuted.push({ action: action.action, entityId: action.campaignId ?? action.adSetId ?? action.adId ?? 'unknown', result, dryRun });
    }

    return { actionsExecuted, analysis };
  }
  ```

- [ ] **Task 2** — Criar `extractAllActions()` para múltiplos JSON no texto (AC: 4)
  ```ts
  function extractAllActions(text: string): Record<string, any>[] {
    const regex = /```json\s*([\s\S]*?)\s*```/g;
    const actions: Record<string, any>[] = [];
    let match;
    while ((match = regex.exec(text)) !== null) {
      try {
        const parsed = JSON.parse(match[1]);
        if (parsed.action) actions.push(parsed);
      } catch { /* skip invalid */ }
    }
    return actions;
  }
  ```

- [ ] **Task 4** — Novo endpoint em `traffic.routes.ts` (AC: 1, 2, 6)
  ```ts
  fastify.post('/atlas/auto-optimize', async (req, reply) => {
    // Auth check
    const authHeader = req.headers.authorization;
    const token = authHeader?.replace('Bearer ', '');
    if (!token || token !== process.env.ATLAS_SCHEDULER_TOKEN) {
      return reply.status(401).send({ error: 'Unauthorized' });
    }

    const { workspaceId, dryRun = true } = req.body as any;

    // Load campaign data
    const trafficService = new TrafficService(workspaceId);
    const [insights, campaigns] = await Promise.race([
      Promise.all([trafficService.getInsights(...), trafficService.getCampaigns(...)]),
      new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), 30000))
    ]) as any;

    const ctx: AtlasContext = { workspace: workspaceId, accountId: ..., insights, campaigns };
    const result = await atlasSchedulerRun(ctx, trafficService, dryRun);
    return reply.send({ ...result, dryRun });
  });
  ```

### GitHub Actions Workflow (para referência futura)

O GitHub Actions que chamará este endpoint ficará em `.github/workflows/atlas-scheduler.yml`:
```yaml
schedule:
  - cron: '0 * * * *'  # toda hora
jobs:
  optimize:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -X POST ${{ secrets.ARIA_API_URL }}/api/traffic/atlas/auto-optimize \
            -H "Authorization: Bearer ${{ secrets.ATLAS_SCHEDULER_TOKEN }}" \
            -H "Content-Type: application/json" \
            -d '{"workspaceId":"default","dryRun":false}'
```
(Este workflow não é parte desta story — apenas referência.)

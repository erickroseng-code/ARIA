# Story 7.3: Google Frontend Reconnection UX

## Status
Ready for Review

## Story

**As a** usuário da Aria,
**I want** que o IntegrationHub mostre o status real da conexão Google em tempo real e permita reconexão sem precisar recarregar a página,
**so that** eu saiba imediatamente quando a integração está ativa e possa reconectar com um clique quando necessário.

## Acceptance Criteria

1. **Auto-refresh periódico:** Quando o IntegrationHub está aberto (`open === true`), o status é atualizado automaticamente a cada 30 segundos via polling.
2. **Refresh pós-OAuth:** Após fechar o popup OAuth, o status é verificado com retry (3 tentativas, intervalo de 1.5s) para garantir que o backend persistiu o token antes de exibir "Conectado".
3. **Estado "Reconectando...":** Enquanto o polling pós-popup está rodando, o botão exibe "Verificando..." com spinner ao invés de "Conectar".
4. **Erro de popup bloqueado:** Se o popup for bloqueado pelo browser, exibe mensagem inline: "Popup bloqueado. Clique aqui para abrir em nova aba." com link direto para a URL OAuth.
5. **Health endpoint:** O status do Google é consultado em `/api/auth/google/health` (novo endpoint da Story 7.2) ao invés de apenas `/api/auth/google/status` — usa o campo `connected` da resposta.

## Scope

**IN:**
- Adicionar polling de 30s em `IntegrationHub.tsx` (quando hub está aberto)
- Melhorar retry pós-popup (3x com 1.5s de intervalo)
- Adicionar estado visual "Verificando..." durante o retry pós-popup
- Adicionar tratamento de popup bloqueado com mensagem clara
- Atualizar `statusUrl` do Google para usar `/api/auth/google/health`

**OUT:**
- Não alterar integrações ClickUp, Notion, Telegram (apenas Google nesta story)
- Não alterar backend (coberto pelas Stories 7.1 e 7.2)
- Não criar novos componentes — tudo dentro do `IntegrationHub.tsx` existente

## Dependencies

- **Story 7.2** deve estar concluída (endpoint `/api/auth/google/health` deve existir)

## Complexity Estimate

**T-Shirt:** Small (S)
**Story Points:** 3
**Dev Time:** ~4 horas

## Business Value

Resolve a percepção de instabilidade: usuário vê "Desconectado" mas na verdade o token está lá — o problema é que o status não estava sendo consultado após reconexão. Com retry pós-popup e polling, o feedback visual estará sempre correto.

## Risks

- **Risco:** Polling de 30s gera requisições desnecessárias ao backend quando hub está aberto por longo tempo
- **Mitigação:** Polling só ocorre quando `open === true` e é cancelado no cleanup do `useEffect`

## Tasks / Subtasks

- [ ] **Task 1** — Atualizar `statusUrl` do Google para usar `/health` (AC: 5)
  - Em `INTEGRATIONS` array, mudar:
    ```ts

    statusUrl: `${API_BASE}/api/auth/google/health`,
    ```
  - Verificar que `fetchStatuses` já usa `data.connected` (já usa — sem mudança adicional)

- [ ] **Task 2** — Polling automático quando hub está aberto (AC: 1)
  - Adicionar segundo `useEffect` para polling:
    ```ts
    useEffect(() => {
      if (!open) return;
      const interval = setInterval(() => {
        fetchStatuses();
      }, 30_000);
      return () => clearInterval(interval);
    }, [open]);
    ```

- [ ] **Task 3** — Retry pós-popup com estado "Verificando..." (AC: 2, 3)
  - Adicionar estado `verifying: string | null` (id da integração sendo verificada)
  - Substituir o `setTimeout(() => fetchStatuses(), 800)` atual por retry com 3 tentativas:
    ```ts
    // Após popup.closed detectado:
    setVerifying(integration.id);
    setConnecting(null);

    let attempts = 0;
    const maxAttempts = 3;
    const retryInterval = setInterval(async () => {
      attempts++;
      await fetchStatuses();
      const currentStatus = statuses[integration.id]; // verificar via ref
      if (currentStatus === true || attempts >= maxAttempts) {
        clearInterval(retryInterval);
        setVerifying(null);
      }
    }, 1500);
    ```
  - **Nota:** `statuses` dentro do closure será stale — usar `useRef` para o status atual ou refatorar para callback de `fetchStatuses`

- [ ] **Task 4** — Estado visual "Verificando..." (AC: 3)
  - No JSX, adicionar condição para `verifying === integration.id`:
    ```tsx
    {verifying === integration.id && (
      <span className="flex items-center gap-1.5 px-3 py-1.5 rounded-full text-xs font-medium bg-blue-500/15 text-blue-400 border border-blue-500/20 flex-shrink-0">
        <Loader2 className="w-3 h-3 animate-spin" />
        Verificando...
      </span>
    )}
    ```

- [ ] **Task 5** — Mensagem de popup bloqueado (AC: 4)
  - No bloco `else` de popup bloqueado (linha ~239 do IntegrationHub.tsx atual):
    ```ts
    } else {
      setConnecting(null);
      setPopupBlocked(integration.id); // novo estado
    }
    ```
  - Adicionar estado `popupBlocked: string | null`
  - No JSX, quando `popupBlocked === integration.id`:
    ```tsx
    <div className="flex-1 text-xs text-amber-400">
      Popup bloqueado.{' '}
      <a href={oauthUrl} target="_blank" rel="noopener noreferrer"
         className="underline hover:text-amber-300">
        Abrir em nova aba
      </a>
    </div>
    ```
  - `oauthUrl` precisa ser armazenado em estado quando fetched

- [ ] **Task 6** — Cleanup correto dos intervalos
  - Garantir que todos os `setInterval` retornam cleanup functions nos useEffect
  - Testar: abrir hub → fechar hub → verificar que nenhum interval continua rodando

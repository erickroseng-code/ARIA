# Story 9.2: Scheduled Task Endpoints

## Status
Ready for Dev

## Story

**As a** GitHub Actions scheduler,
**I want** chamar endpoints autenticados da Aria para disparar tarefas agendadas,
**so that** a Aria execute ciclos de otimização e relatórios automaticamente no horário programado.

## Acceptance Criteria

1. **Auth:** Todos os endpoints de tarefas verificam `Authorization: Bearer SCHEDULER_SECRET`. Requests sem token válido → 401.
2. **Endpoint weekly-report:** `POST /api/tasks/weekly-report` busca insights de campanhas do Atlas (últimos 7 dias) e envia um relatório formatado via Telegram para `TELEGRAM_CHAT_ID`.
3. **Endpoint atlas-optimize:** `POST /api/tasks/atlas-optimize` é um alias que chama internamente o mesmo fluxo de `atlasSchedulerRun()` já implementado na Story 8.2. Aceita body `{ workspaceId, accountId, dryRun? }`.
4. **Endpoint maverick-research:** `POST /api/tasks/maverick-research` dispara uma análise de trends do Maverick (se disponível) e envia resultado via Telegram. Se Maverick não estiver configurado, retorna `{ skipped: true, reason: 'Maverick not configured' }`.
5. **Resposta padronizada:** Todos os endpoints retornam `{ task, executedAt, success, result }`.
6. **Sem bloquear:** Todos os endpoints respondem imediatamente após iniciar a tarefa (fire-and-forget para operações longas) ou após conclusão para operações curtas (< 10s).

## Scope

**IN:**
- Novo arquivo `aria/apps/api/src/routes/scheduled-tasks.routes.ts`
- Registrar com prefix `/api/tasks` em `server.ts`
- Adicionar `SCHEDULER_SECRET` ao `.env`
- O `atlas-optimize` reutiliza `atlasSchedulerRun()` do módulo traffic

**OUT:**
- Não duplicar lógica de `POST /api/traffic/atlas/auto-optimize` — reutilizar
- Não implementar Maverick com lógica complexa — apenas stub que verifica se existe
- Não alterar `ScheduledReportService` existente (é para outra funcionalidade)

## Dependencies

- **Story 9.1** — `sendTelegram()` deve estar centralizado em `shared/telegram.ts`
- **Story 8.2** — `atlasSchedulerRun()` deve existir (já implementado)

## Complexity Estimate

**T-Shirt:** Small (S)
**Story Points:** 3
**Dev Time:** ~3 horas

## Business Value

Sem estes endpoints, os GitHub Actions não têm como disparar as tarefas. É a ponte entre o cron externo e a lógica interna da Aria.

## Risks

- **Risco:** `weekly-report` pode demorar se a Meta API estiver lenta
- **Mitigação:** Timeout de 30s na chamada de dados, responder 200 imediatamente e processar assíncrono

  ```ts
    const token = req.headers.authorization?.replace('Bearer ', '');
    if (!token || token !== process.env.SCHEDULER_SECRET) {
  ```


Isso permite o GitHub Actions chamar sem precisar passar esses parâmetros no body.

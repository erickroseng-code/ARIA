# Story 7.1: Google Token Persistence & Auto-Refresh

## Status
Ready for Review

## Story

**As a** sistema Aria em produção,
**I want** que os tokens Google sejam persistidos corretamente e renovados automaticamente antes de expirarem,
**so that** as integrações com Calendar, Drive e Gmail funcionem sem interrupção mesmo após horas de operação.

## Acceptance Criteria

1. **Bug Fix — setCredentials duplo:** `WorkspaceClient.ts` chama `setCredentials` duas vezes (linhas 55-56), fazendo a segunda chamada sobrescrever a primeira e perder o `refresh_token`. A correção deve usar uma única chamada combinando ambos os tokens.
2. **Token listener:** Após a biblioteca `googleapis` renovar automaticamente o access_token, o novo token é salvo de volta no SQLite via listener `auth.on('tokens', ...)`.
3. **Fallback preservado:** Quando apenas `GOOGLE_REFRESH_TOKEN` está no `.env` (sem DB), o client configura corretamente `{ refresh_token }` e a renovação automática funciona.
4. **Teste de renovação:** Existe teste unitário que verifica que, após `invalid_grant`, o callback `notifyInvalidGrant` é chamado e o token é marcado como inválido no DB.
5. **Sem breaking changes:** `CalendarService`, `DriveService`, `GmailService`, `SheetsService` e `DocsService` continuam funcionando — nenhuma assinatura pública é alterada.

## Scope

**IN:**
- Corrigir bug em `WorkspaceClient.ts` (`createWorkspaceClient` — linhas 55-56)
- Adicionar listener `auth.on('tokens', callback)` para persistir access_token renovado no SQLite
- O callback de persistência é injetado via `setWorkspaceTokenPersistor(cb)` (nova função exportada)
- Configurar o persistor em `server.ts` (ao lado de `setWorkspaceTokenResolver`)

**OUT:**
- Não alterar o fluxo OAuth de autorização inicial (`/url`, `/callback`)
- Não alterar schemas do banco de dados
- Não alterar o frontend

## Dev Notes

### Bug identificado (source: leitura direta do código)

**Arquivo:** `aria/packages/integrations/src/google-workspace/WorkspaceClient.ts` — linhas 55-56

```ts
// BUG ATUAL:
if (refreshToken) auth.setCredentials({ refresh_token: refreshToken }); // seta refresh_token
if (accessToken) auth.setCredentials({ access_token: accessToken });    // SOBRESCREVE tudo — refresh_token perdido!

// CORREÇÃO:
auth.setCredentials({
  refresh_token: refreshToken ?? undefined,
  access_token: accessToken ?? undefined,
});
```

**Por que quebra:** `OAuth2Client.setCredentials()` substitui o objeto `credentials` inteiro, não faz merge. Fonte: googleapis README e código-fonte de `google-auth-library`.

### Token listener (googleapis)

O `OAuth2Client` emite evento `'tokens'` sempre que obtém um novo access_token (via refresh implícito em chamadas de API ou via `refreshAccessToken()`). Documentação: googleapis/google-auth-library-nodejs.

```ts
auth.on('tokens', (tokens) => {
  // tokens.access_token — novo token
  // tokens.expiry_date — timestamp de expiração em ms
  // tokens.refresh_token — só presente se Google emitiu novo refresh_token (raro)
});
```

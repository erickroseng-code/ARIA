## Story

**As a** usuário,
**I want** usar linguagem natural para criar tarefas sem seguir sintaxe rígida,
**so that** eu possa criar tarefas de forma rápida e intuitiva, conversando naturalmente com ARIA.

## Acceptance Criteria

1. Claude API extrai entidades em português livre: título, data, hora, cliente, prioridade, destino
2. Datas relativas interpretadas inteligentemente: "amanhã", "sexta", "semana que vem", "próxima segunda à tarde", "daqui a 3 dias às 10h"
3. Quando destino não especificado: ARIA pergunta "Qual destino? (ClickUp, Notion, ou em ambos?)" e aguarda resposta indefinidamente
4. Quando data não especificada: tarefa criada sem prazo com confirmação: "Vou criar sem data de vencimento. Tá certo?"
5. Nome de cliente detectado → ARIA faz fuzzy matching e oferece confirmação: "Você quis dizer: [Cliente A], [Cliente B], [Cliente C]?"
6. Confidence baixo OU campos críticos faltando → confirmação obrigatória antes de criar
7. Confirmação é interativa: usuário responde com "sim", "não", ou alterna as opções oferecidas
8. Mensagem de sucesso: "✅ Tarefa criada: *{título}* — {destino} | Data: {dueDateFormatted} | Cliente: {clientName}"

## Complexity Estimate

**T-Shirt Sizing:** Medium-High (M+)
**Story Points (if using):** 10-15 points
**Estimated Dev Time:** 4-5 days
**Estimated QA Time:** 1-2 days

**Complexity Drivers:**
- Natural language processing with Claude API (entity extraction)
- Intelligent date parsing (relative dates + time + timezone)
- Fuzzy matching for client names (Levenshtein distance)
- Multi-step confirmation flow with state management
- Integration with Stories 3.2 + 3.3 (ClickUp + Notion destinations)
- Free-form validation pattern (less structured than typical API)

## Risks & Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|-----------|
| **Claude API parsing fails** | Entities not extracted or wrong extraction | Medium | Add unit tests for 20+ natural language samples; log raw extraction for debugging; fallback to structured prompt if confidence < 0.5 |
| **Timezone/Date Parsing Errors** | User sees wrong date ("tomorrow" becomes wrong date) | Medium | Use `date-fns` library with explicit PT-BR locale; document timezone assumptions; test with DST edge cases |
| **Fuzzy Matching Too Loose** | Wrong client selected (user says "João", matches "João Pedro" + "João Silva") | Medium | Use Levenshtein distance with threshold 0.75+; always show ALL matches to user; don't auto-select |
| **Confirmation Flow Ambiguity** | User confused by multiple questions ("What's the date? What's the client? What's the destination?") | Medium | Ask one question at a time; offer quick buttons (Telegram: inline buttons; Web: radio buttons) |
| **Infinite Wait for Response** | User never responds to confirmation, task never created | Low | Document behavior in user guide; add chat indicator "Waiting for your response..." |
| **Entity Extraction Hallucination** | Claude invents data not in user message (ex: user says "task", Claude adds random date) | Medium | Strict validation: only use entities that are explicitly present or inferrable; log all extractions |

## Out of Scope

The following features are **NOT** included in this story:

- ❌ **Multi-language support** — Portuguese only (Story X.X for i18n)

- ❌ **History-based inference** — No context carryover from previous messages

## Dev Notes

### Natural Language Processing Strategy

**Entity Extraction Flow:**
1. User sends message: `"cria follow-up com João para sexta"`
2. `TaskEntityExtractor.extract()` called with Claude Haiku
3. Prompt instructs Claude to extract as JSON-like structure (not strict JSON, but readable):
   ```
   title: follow-up
   client: João
   dueDate: (próxima sexta)
   priority: (não especificado)
   destination: (não especificado)
   confidence: 0.85
   ```
4. Code parses response and validates each field
5. If confidence < 0.7 or required fields missing → confirmation flow

**Key Decision: Free-Form Extraction**
- Claude extracts naturally, code validates → more flexible, less hallucination
- NOT using Claude's structured output (JSON mode) to allow more natural parsing

### Date Parsing Implementation

**Relative Dates (using `date-fns`):**
```typescript
"amanhã" → addDays(today, 1)
"próxima segunda" → nextMonday(today)
"semana que vem" → addDays(today, 7)
"daqui a 3 dias" → addDays(today, 3)
"segunda à tarde" → nextMonday(today) + 14:00 (14h default)
```

**Time Parsing:**
- "15h" → 15:00
- "14:30" → 14:30
- "fim de expediente" → 17:00 (default)
- If time without date → use today

**Timezone:**
- Always convert to `America/Sao_Paulo` (PT-BR)
- Store as ISO 8601 UTC in database
- Format for display: `new Intl.DateTimeFormat('pt-BR', {...})`

### Client Fuzzy Matching

**Algorithm:** Levenshtein distance (via `string-similarity` npm package)

```typescript
const threshold = 0.75;
const matches = availableClients
  .map(c => ({ client: c, similarity: stringSimilarity(input, c.name) }))
  .filter(m => m.similarity >= threshold)
  .sort((a, b) => b.similarity - a.similarity)
  .slice(0, 3); // Top 3
```

**Confidence Scoring:**
- 1.0 = exact match (case-insensitive) → no confirmation needed
- 0.85-0.99 = high similarity → offer choice
- 0.75-0.84 = moderate similarity → offer choice with note

### Confirmation Flow State Management

**Challenge:** Multi-step confirmations need state tracking across messages

**Solution:** Extend `SessionContext` in `packages/shared/src/types/session.types.ts`:
```typescript
interface SessionContext {
  // ... existing fields
  pendingConfirmation?: {
    entityType: 'date' | 'client' | 'destination';
    entities: ExtractedEntities;
    options?: string[]; // For client matches: ["João", "João Silva"]
    createdAt: Date;
  };
}
```

- When confirmation needed: set `pendingConfirmation`
- Next user message: check if `pendingConfirmation` exists
- If yes: parse as confirmation response, not new command
- If no: proceed with normal parsing

### Error Codes for Natural Language Tasks

| Code | Trigger | Message |
|------|---------|---------|
| `TASK_001` | Extraction failed (Claude error) | "Não consegui entender. Pode tentar novamente?" |
| `TASK_002` | Confidence too low (<0.5) | "Comando muito vago. Dê mais detalhes." |
| `TASK_003` | Client not found (after fuzzy match) | "Nenhum cliente similar encontrado." |
| `TASK_004` | Invalid date format | "Data não reconhecida. Tente: 'amanhã', 'próxima sexta'" |
| `TASK_005` | Confirmation timeout (with explicit timeout) | "Confirmação expirou. Tente novamente." |

### Testing Notes

**Critical Paths to Test:**
1. Happy path: all entities present, high confidence → immediate creation
2. Falta data: missing date, ask for confirmation → create without due date
3. Cliente ambíguo: multiple fuzzy matches → user selects from list
4. Confiança baixa: low confidence in extraction → ask to rephrase
5. Múltiplas confirmações: need both client confirmation AND destination → handle sequentially

**Edge Cases:**
- "próxima segunda" on a Monday → should be 8 days away, not tomorrow
- DST transition: "amanhã" on DST day → correct hour offset
- Leap year: "daqui a 365 dias" from Feb 28
- Special chars in client names: "João's Consultoria", "Empresa (Beta)", "A&B Corp"

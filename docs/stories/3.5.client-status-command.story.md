## Story

**As a** usuário,
**I want** consultar o status consolidado de um cliente rapidamente,
**so that** eu possa ter uma visão 360° do cliente com uma única mensagem.

## Acceptance Criteria

1. Comando reconhecido em variações: "status do cliente X", "resumo da Empresa X", "como está João?", "o que sei sobre X?"
2. Dados consolidados retornados: tarefas abertas no ClickUp, últimas anotações no Notion, Planos de Ataque criados, próximas reuniões (Google Calendar)
3. Resposta formatada com emojis visuais: ✅ concluído, 🔴 atrasado, 📅 agendado, 📌 importante
4. Resposta entregue no mesmo canal onde o comando foi feito (Telegram ou Web)
5. Link direto para o perfil completo do cliente no Notion incluído na resposta
6. Se cliente não encontrado: "Qual cliente? (João, Empresa X, ...)" com lista de clientes disponíveis
7. Se nenhuma atividade encontrada: "Sem atividades registradas para {cliente}. Crie a primeira tarefa!"
8. Resposta gerada em menos de 3 segundos (performance critical)

## Risks & Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|-----------|
| **Slow API calls block response** | Exceeds 3s SLA, poor UX | Medium | Implement parallel fetching (`Promise.all`), cache results for 5 min, timeout individual APIs at 1.5s |
| **Missing data from one service** | Incomplete status response | Low | Graceful degradation: show what's available, note if ClickUp/Calendar unavailable |
| **Client not found (fuzzy match fails)** | User confused, unclear error | Low | Use same fuzzy matching from Story 3.4; show top 3 suggestions |
| **Notion database schema mismatch** | Can't fetch plan data | Low | Validate schema at startup; log errors; document required properties |
| **Google Calendar auth expired** | Calendar data unavailable | Medium | Check token validity; refresh if expired; fallback to Notion schedule |

## Out of Scope

- ❌ **Historical analytics** — Only current status, no trends
- ❌ **Custom report generation** — Fixed format only
- ❌ **Client comparison** — Single client only, not multiple
- ❌ **Scheduled status reports** — Manual command only
- ❌ **Subtask breakdown** — High-level status only (open/closed counts)

## Dev Notes

### Data Aggregation Strategy

**Parallelization with `Promise.all`:**

```typescript
const [tasks, notes, plan, meeting] = await Promise.all([
  clickupService.getOpenTasks(clientId),
  notionService.getRecentNotes(clientId),
  notionService.getPlanOfAttack(clientId),
  googleCalendarService.getNextEvent(clientId),
]);
```

**Individual Timeouts (1.5s each):**
```typescript
const withTimeout = (promise, ms) =>
  Promise.race([
    promise,
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error('Timeout')), ms)
    ),
  ]);

// Each API call is wrapped
const tasks = await withTimeout(clickupService.getOpenTasks(clientId), 1500);
```

### Caching Strategy

**In-Memory Cache (5 min TTL):**
```typescript
private cache = new Map<string, { data: ClientStatus; expiry: Date }>();

getClientStatus(clientId: string): Promise<ClientStatus> {
  const cached = this.cache.get(clientId);
  if (cached && cached.expiry > new Date()) {
    return Promise.resolve(cached.data);
  }
  // Fetch fresh data
  const status = await this.fetchStatus(clientId);
  this.cache.set(clientId, { data: status, expiry: Date.now() + 5 * 60 * 1000 });
  return status;
}
```

**OR Redis Cache (if available):**
- Key: `client:status:{clientId}`
- TTL: 300s
- Fallback to in-memory if Redis unavailable

### Performance Requirements

- **Target:** < 3 seconds end-to-end
- **Budget Breakdown:**
  - Network calls: 1.5s (parallel, with 1s timeout each)
  - Data processing: 0.5s
  - Formatting: 0.5s
  - Buffer: 0.5s

**Monitoring:**
- Log response time for each request
- Alert if > 2.5s (warning), > 3s (critical)
- Track which API is slowest for optimization

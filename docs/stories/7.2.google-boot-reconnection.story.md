# Story 7.2: Google Boot-Time Reconnection & Health Check

## Status
Ready for Review

## Story

**As a** operador da Aria em produção,
**I want** que a API valide a conexão Google no startup e exponha um endpoint de saúde dedicado,
**so that** eu possa monitorar o status da integração e ser alertado se os tokens falharem antes de qualquer chamada de usuário.

## Acceptance Criteria

1. **Boot validation:** Após o servidor iniciar, uma verificação assíncrona não-bloqueante testa os tokens Google chamando `isWorkspaceConfigured()` e tentando obter um access_token válido. Resultado é logado com `console.log`.
2. **Health endpoint:** `GET /health/google` retorna `{ connected: boolean, source: 'db' | 'env' | null, lastChecked: string }` com HTTP 200 (connected) ou 503 (disconnected).
3. **Health integrado:** O endpoint `GET /health` existente inclui `google: 'connected' | 'disconnected' | 'not_configured'` no objeto `services`.
4. **Reconnection log:** Quando tokens são inválidos no startup, o log imprime instruções claras: `[Google] Tokens inválidos ou ausentes. Re-autorize em /api/auth/google/url`.
5. **Não-bloqueante:** A validação de boot não atrasa a inicialização do servidor. Se falhar, o servidor sobe normalmente com status `google: 'disconnected'`.

## Scope

**IN:**
- Adicionar verificação assíncrona pós-boot em `server.ts` (após `fastify.listen`)
- Criar endpoint `GET /api/health/google` em `server.ts` ou novo arquivo de routes
- Atualizar `GET /health` para incluir status Google
- Usar `isWorkspaceConfigured()` e `createWorkspaceClient()` já existentes

**OUT:**
- Não alterar lógica de autenticação OAuth
- Não criar nova tabela no banco de dados
- Não alterar `WorkspaceClient.ts` (coberto pela Story 7.1)

## Dependencies

- **Story 7.1** deve estar concluída (bug fix de `setCredentials`) para que a verificação de boot seja confiável

## Complexity Estimate

**T-Shirt:** Small (S)
**Story Points:** 2
**Dev Time:** ~3 horas

## Business Value

Visibilidade crítica para produção: sem este endpoint, não há como o UptimeRobot ou GitHub Actions saber se a integração Google está ativa antes de tentar usá-la.

## Risks

- **Risco:** Verificação de boot faz chamada à API Google e pode adicionar latência no startup
- **Mitigação:** Usar `isWorkspaceConfigured()` (sem chamada de rede) para verificação rápida; a verificação de token válido é assíncrona e não-bloqueante

  - Lógica:
    ```ts
    async function checkGoogleHealth(): Promise<{ connected: boolean; source: string | null }> {
      try {
        const configured = await isWorkspaceConfigured();
        if (!configured) return { connected: false, source: null };
        // Tenta criar cliente — vai falhar com invalid_grant se tokens ruins
        await createWorkspaceClient();

        const row = stmt.get('google') as any;
        return { connected: true, source: row ? 'db' : 'env' };
      } catch {
        return { connected: false, source: null };
      }
    }
    ```

## Dev Notes

### Estrutura atual de `server.ts`

O endpoint `/health` (linha 123-151) já monta objeto `services` com `clickup` e `chat`. Adicionar `google` seguindo o mesmo padrão.

`isWorkspaceConfigured()` e `createWorkspaceClient()` já são importáveis de `@aria/integrations` — verificar se estão re-exportados em `aria/packages/integrations/src/index.ts`.

### Localização dos arquivos

```
aria/apps/api/src/
├── server.ts                          ← MODIFICAR (boot validation + health services.google)
└── routes/
    ├── google-auth.routes.ts          ← MODIFICAR (adicionar GET /health)
    └── __tests__/
        └── google-auth.health.test.ts ← CRIAR
```

### Nota sobre `createWorkspaceClient()` no health check

`createWorkspaceClient()` não faz chamada de rede — apenas instancia `OAuth2Client` com credenciais. Portanto a verificação de boot é rápida. A validação de que os tokens _realmente funcionam_ (vs. apenas existem) exigiria uma chamada real à API (ex.: `calendar.calendarList.list()`), o que é opcional e fora do escopo desta story.

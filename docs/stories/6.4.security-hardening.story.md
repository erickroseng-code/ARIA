## Story

**As a** sistema,
**I want** estar hardened contra ataques comuns e monitorado 24/7,
**so that** dados dos usuários estejam seguros e vulnerabilidades sejam detectadas rapidamente.

## Acceptance Criteria

1. Rate limiting: 100 req/min por IP global, 10 req/min por usuário autenticado
2. Security headers: HSTS, CSP, X-Frame-Options, X-Content-Type-Options
3. Input validation: Sanitização de todo input (SQL injection, XSS, LDAP injection)
4. Authentication: JWT com refresh tokens, no-stored passwords in logs
5. Authorization: Role-based access control (RBAC), verificação em cada endpoint
6. API keys: Rotação mensal, auditoria de uso
7. Database: Prepared statements (Prisma ORM), no raw SQL, encrypted sensitive fields
8. Secrets management: Não commitadas no Git, rotação via GitHub Secrets
9. CORS: Configurado para domínios específicos apenas
10. Logging: Auditoria de ações de usuários, sem sensible data (passwords, tokens)
11. Monitoring: Alerts para suspicious activity (failed logins, rate limit violations)
12. Penetration test: Relatório de security audit incluído

## Risks & Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|-----------|
| **Rate limiter too strict** | Legitimate users blocked | Low | Use generous limits; whitelist certain endpoints |
| **CSP too strict** | UI breaks (images, scripts not load) | Medium | Test all resources; use nonce for inline scripts |
| **Encryption key lost** | Cannot decrypt sensitive data | Low | Store keys in secure manager; backup procedure |
| **Secrets leaked in logs** | Security breach | Low | Use sanitization; no passwords in error messages |
| **Monitoring generates false positives** | Alert fatigue | Medium | Tune thresholds; test alerts in staging |
| **Performance impact** | Slow API responses | Low | Cache rate limit checks; optimize queries |

## Tasks / Subtasks

- [ ] Task 1: Rate limiting (AC: 1)

- [ ] Task 2: Security headers (AC: 2)

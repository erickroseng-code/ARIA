# Story 9.3: GitHub Actions — Scheduler Workflows

## Status
Ready for Dev

## Story

**As a** sistema autônomo da Aria,
**I want** que o GitHub Actions dispare tarefas agendadas em horários fixos,
**so that** a Aria execute otimizações e relatórios semanais sem nenhuma intervenção manual.

## Acceptance Criteria

1. **Workflow weekly-report:** `.github/workflows/aria-weekly-report.yml` roda toda segunda-feira às 09:00 (UTC-3 = 12:00 UTC) e chama `POST /api/tasks/weekly-report`.
2. **Workflow atlas-optimize:** `.github/workflows/aria-atlas-optimize.yml` roda diariamente às 22:00 (UTC-3 = 01:00 UTC próximo dia) e chama `POST /api/tasks/atlas-optimize`.
3. **Autenticação:** Todos os workflows usam `secrets.ARIA_API_URL` e `secrets.SCHEDULER_SECRET` — nenhum valor hardcoded.
4. **Falha tolerante:** Workflows não falham silenciosamente — status do curl é verificado e o job falha se o endpoint retornar erro HTTP.
5. **Disparo manual:** Todos os workflows têm `workflow_dispatch` para permitir execução manual pelo GitHub UI.
6. **Secrets necessários documentados:** README ou comentário no workflow lista os secrets a configurar no repositório GitHub.

## Scope

**IN:**
- Criar `.github/workflows/aria-weekly-report.yml`
- Criar `.github/workflows/aria-atlas-optimize.yml`
- Secrets a configurar no GitHub: `ARIA_API_URL`, `SCHEDULER_SECRET`

**OUT:**
- Não configurar os secrets (isso é manual no GitHub Settings)
- Não criar workflow para Maverick (Story 9.2 deixa como stub — ativar após Epic 10)
- Não usar GitHub Actions para CI/CD (workflows são apenas schedulers HTTP)

## Dependencies

- **Story 9.2** deve estar concluída (endpoints devem existir antes dos workflows)
- **Epic 11** deve estar concluída (Aria precisa estar no Render com URL pública)
  - *Nota:* Os arquivos podem ser criados agora, mas só funcionarão após o deploy

## Tasks / Subtasks

- [ ] **Task 1** — Criar `.github/workflows/aria-weekly-report.yml` (AC: 1, 3, 4, 5)
  ```yaml
  name: Aria — Weekly Report

  on:
    schedule:
      - cron: '0 12 * * 1'  # toda segunda-feira às 09:00 BRT (12:00 UTC)
    workflow_dispatch:       # disparo manual pelo GitHub UI

  jobs:
    weekly-report:
      runs-on: ubuntu-latest
      steps:
        - name: Trigger weekly report
          run: |
            HTTP_STATUS=$(curl -s -o /tmp/response.json -w "%{http_code}" \
              -X POST "${{ secrets.ARIA_API_URL }}/api/tasks/weekly-report" \
              -H "Authorization: Bearer ${{ secrets.SCHEDULER_SECRET }}" \
              -H "Content-Type: application/json" \
              -d '{}')

            echo "Response: $(cat /tmp/response.json)"
            echo "HTTP Status: $HTTP_STATUS"

            if [ "$HTTP_STATUS" -lt 200 ] || [ "$HTTP_STATUS" -ge 300 ]; then
              echo "::error::Weekly report endpoint returned HTTP $HTTP_STATUS"
              exit 1
            fi
  ```

- [ ] **Task 2** — Criar `.github/workflows/aria-atlas-optimize.yml` (AC: 2, 3, 4, 5)
  ```yaml
  name: Aria — Atlas Daily Optimization

  on:
    schedule:
      - cron: '0 1 * * *'   # diariamente às 22:00 BRT (01:00 UTC próximo dia)
    workflow_dispatch:

  jobs:
    atlas-optimize:
      runs-on: ubuntu-latest
      steps:
        - name: Trigger Atlas optimization
          run: |
            HTTP_STATUS=$(curl -s -o /tmp/response.json -w "%{http_code}" \
              -X POST "${{ secrets.ARIA_API_URL }}/api/tasks/atlas-optimize" \
              -H "Authorization: Bearer ${{ secrets.SCHEDULER_SECRET }}" \
              -H "Content-Type: application/json" \
              -d '{"dryRun": false}')

            echo "Response: $(cat /tmp/response.json)"
            echo "HTTP Status: $HTTP_STATUS"

            if [ "$HTTP_STATUS" -lt 200 ] || [ "$HTTP_STATUS" -ge 300 ]; then
              echo "::error::Atlas optimize endpoint returned HTTP $HTTP_STATUS"
              exit 1
            fi
  ```
  - **Nota:** `dryRun: false` — quando ativar, garantir que `ATLAS_WRITE_ENABLED=true` no servidor

## Dev Notes

### Como configurar os secrets no GitHub

Após deploy no Render:
1. Ir em `github.com/{repo}/settings/secrets/actions`
2. Adicionar `ARIA_API_URL` = URL do Render (ex: `https://aria-api.onrender.com`)
3. Adicionar `SCHEDULER_SECRET` = valor de `SCHEDULER_SECRET` do `.env`

### Verificar fuso horário

GitHub Actions usa UTC. Horários:
- 09:00 BRT = 12:00 UTC → `cron: '0 12 * * 1'`
- 22:00 BRT = 01:00 UTC (dia seguinte) → `cron: '0 1 * * *'`

Durante horário de verão (BRST = UTC-2):
- 09:00 BRST = 11:00 UTC → ajustar manualmente se necessário

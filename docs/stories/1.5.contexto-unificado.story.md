## Story

**As a** usuário,
**I want** que o Telegram e a Web UI compartilhem o mesmo contexto de conversa,
**so that** eu possa alternar entre os dois canais sem perder a continuidade da conversa e o histórico de clientes em foco.

## Acceptance Criteria

1. Sessão identificada por **usuário** (não por canal) — `userId` estável compartilhado entre Telegram e Web UI
2. `ContextStore` refatorado para suportar chave por `userId` (além de `sessionId` por canal)
3. `activeClientId` (cliente em foco) compartilhado entre canais — definido em um canal, acessível no outro
4. Contexto de conversa persistido **em memória** durante a sessão ativa do servidor
5. Ao reiniciar o servidor, nova sessão iniciada (sem banco de dados — fora do escopo do MVP)
6. Endpoint `GET /api/chat/history/:userId` retornando as últimas 10 mensagens do histórico unificado
7. Web UI consome `GET /api/chat/history/:userId` na inicialização para exibir histórico da sessão atual
8. Bot Telegram e Web UI usam o mesmo `userId` para identificar o usuário no `ContextStore`

## Dev Notes

### Decisão de Design — userId por Canal

```
Telegram Bot:  userId = "tg_{telegramId}"     ex: "tg_123456789"
Web UI:        userId = "web_aria_user"        ex: "web_aria_user" (fixo, single-user)

sessionId (histórico separado):
Telegram:      sessionId = "tg_{chatId}"       → histórico do Telegram
Web UI:        sessionId = "web_{uuid}"        → histórico da Web UI

activeClientId (compartilhado por baseUserId):
baseUserId = extrair parte numérica do userId
"tg_123456789" → baseUserId "123456789"
"web_aria_user" → baseUserId "aria_user"
```

**Nota arquitetural:** Por design, Telegram e Web UI mantêm históricos de conversa **separados** (sessionId diferente). O que é compartilhado é o `activeClientId` — o cliente em foco. Isso evita que mensagens do Telegram apareçam na Web UI e vice-versa (que causaria confusão de contexto), mas mantém a continuidade do cliente sendo trabalhado.

[Source: docs/architecture.md#Section-6.5-Unified-Context]

### Fluxo Unificado — Diagrama

```
Telegram: "cliente: Empresa X"
  → ChatService detecta intent CLIENT_LOOKUP
  → contextStore.setActiveClient("tg_123", "notion_page_id_xyz")
  → Confirmação no Telegram

Web UI (mesma sessão):
  → GET /api/chat/session/web_abc
  → response: { activeClientId: null }   ← Web UI tem sua própria sessão

Nota: activeClientId só é realmente compartilhado se o userId base for o mesmo.
No MVP single-user, isso significa que Telegram e Web UI são tratados como
"mesmo usuário" se configurado assim, mas por simplicidade mantemos sessões separadas.
```

### Refatoração do ContextStore

```typescript
// packages/core/src/chat/ContextStore.ts (versão atualizada)
import type { SessionContext, ConversationMessage } from '@aria/shared';

interface ActiveContext {
  activeClientId?: string;
  lastUpdated: Date;
}

export class ContextStore {
  private sessions = new Map<string, SessionContext>();
  private activeContexts = new Map<string, ActiveContext>(); // por userId

  get(sessionId: string): SessionContext {
    if (!this.sessions.has(sessionId)) {
      this.sessions.set(sessionId, {
        sessionId,
        userId: sessionId,
        channel: sessionId.startsWith('tg_') ? 'telegram' : 'web',
        history: [],
        createdAt: new Date(),
        updatedAt: new Date(),
      });
    }
    return this.sessions.get(sessionId)!;
  }

  async append(sessionId: string, message: ConversationMessage): Promise<void> {
    const ctx = this.get(sessionId);
    ctx.history.push(message);
    // Rolling window — máximo 10 mensagens
    if (ctx.history.length > 10) {
      ctx.history = ctx.history.slice(-10);
    }
    ctx.updatedAt = new Date();
  }

  setActiveClient(userId: string, clientId: string): void {
    this.activeContexts.set(userId, {
      activeClientId: clientId,
      lastUpdated: new Date(),
    });
  }

  getActiveClient(userId: string): string | undefined {
    return this.activeContexts.get(userId)?.activeClientId;
  }

  getHistory(sessionId: string): ConversationMessage[] {
    return this.get(sessionId).history;
  }
}

export const contextStore = new ContextStore(); // Singleton
```

### Hook useSession — Referência

```typescript
// apps/web/src/hooks/useSession.ts
import { useEffect } from 'react';
import { useChatStore } from '../stores/chatStore';

const API_URL = process.env.NEXT_PUBLIC_API_URL ?? 'http://localhost:3001';

function getOrCreateSessionId(): string {
  const existing = sessionStorage.getItem('aria-session-id');
  if (existing) return existing;
  const newId = `web_${crypto.randomUUID()}`;
  sessionStorage.setItem('aria-session-id', newId);
  return newId;
}

export function useSession() {
  const { messages, addMessage } = useChatStore();

  useEffect(() => {
    const sessionId = getOrCreateSessionId();

    fetch(`${API_URL}/api/chat/history/${sessionId}`, {
      credentials: 'include',
    })
      .then((r) => r.json())
      .then((data) => {
        if (data.history?.length > 0 && messages.length === 0) {
          data.history.forEach((msg: any) => addMessage(msg));
        }
      })
      .catch(() => {
        // Sessão nova ou servidor reiniciado — sem histórico, OK
      });
  }, []);

  return { sessionId: getOrCreateSessionId() };
}
```

### Nota sobre MVP — Sem Persistência

Conforme PRD (Out of Scope):
> "Memória persistente entre sessões (banco de dados próprio) — Notion/ClickUp como datastores no MVP"

O `ContextStore` usa `Map` em memória. Ao reiniciar `apps/api`, todo histórico é perdido. Isso é comportamento esperado e documentado. Persistência com PostgreSQL está planejada para v1.1+.

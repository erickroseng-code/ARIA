# Story 11.2: Render Service Configuration

## Status
Ready

## Story

**As a** operador do sistema Aria,
**I want** configurar o Render como ambiente de deploy da API via Infrastructure as Code,
**so that** o deploy seja reproduzível, controlado por versão, e auto-deploy ocorra a cada push na branch main.

## Acceptance Criteria

1. **`render.yaml`** na raiz do monorepo define o Web Service da API com:
   - `name: aria-api`
   - `type: web`
   - `env: docker`
   - `dockerfilePath: aria/apps/api/Dockerfile`
   - `dockerContext: aria/apps/api`
   - `plan: free`
   - `branch: master`
   - `autoDeploy: true`
2. **Health check** configurado em `/health` com timeout de 60s no `render.yaml`.
3. **Endpoint `/health`** existe na API retornando `{ status: "ok", timestamp: ISO8601, version: "1.0.0" }` com HTTP 200.
4. **Environment variables** mapeadas no `render.yaml` via `envVars` com `sync: false` (não expõe valores, referencia Render Environment Group).
5. **GitHub Actions secret `ARIA_API_URL`** documentado no README de deploy com o valor final `https://aria-api.onrender.com`.
6. **Deploy inicial funciona:** após primeiro push, `https://aria-api.onrender.com/health` retorna 200.

## Scope

**IN:**
- Criar `render.yaml` na raiz do monorepo
- Adicionar endpoint `/health` em `aria/apps/api/src/server.ts`
- Documentar variáveis de ambiente necessárias no Render

**OUT:**
- Não configurar UptimeRobot ainda (Story 11.3)
- Não incluir o frontend Web no deploy (apenas a API)
- Não configurar domínio customizado

## Dependencies

- Story 11.1 concluída: Dockerfile existente e testado

## Business Value

O `render.yaml` elimina a necessidade de configuração manual via UI do Render a cada mudança. O auto-deploy garante que o ambiente de produção sempre reflete o estado atual da branch main.

## Dev Notes

### Render Free Tier Limitações
- **750h/mês** — suficiente para 1 serviço rodando 24/7 (744h)
- **Spin-down após 15 min de inatividade** — mitigado por UptimeRobot na Story 11.3
- **Build timeout:** 15 min — Docker multi-stage build deve ser < 10 min
- **Persistent disk:** não disponível no free tier → banco SQLite precisa de solução alternativa (Turso/libsql para produção real, ou aceitar reset a cada deploy)

### SQLite em Produção
O free tier do Render não tem persistent disk. Opções:
1. **Turso** (libsql cloud) — já integrado no código via `@libsql/client`
2. **Aceitar reset a cada deploy** — OK para dados de análise (Maverick recria)
3. **Upgrade para Render Starter** ($7/mês) — persistent disk

Para MVP: aceitar reset + usar Turso para dados críticos (configurar `DATABASE_URL=libsql://...`).

# Story 11.3: Keep-Alive & Monitoring

## Status
Ready

## Story

**As a** operador do sistema Aria,
**I want** que a API no Render seja mantida ativa continuamente e que alertas sejam enviados ao Telegram quando houver downtime,
**so that** os GitHub Actions schedulers possam chamar a API a qualquer momento sem cold start de 30s.

## Acceptance Criteria

1. **UptimeRobot configurado** (free tier) com monitor HTTP(S) pingando `https://aria-api.onrender.com/health` a cada 5 minutos.
2. **Alerta Telegram** configurado no UptimeRobot: quando downtime > 5 min, mensagem enviada para o chat configurado com texto: `⚠️ [Aria API] Downtime detectado em https://aria-api.onrender.com`.
3. **GitHub Actions secret `ARIA_API_URL`** adicionado ao repositório com o valor `https://aria-api.onrender.com`.
4. **Smoke test no GitHub Actions:** workflow de smoke test que:
   - Faz `curl ${{ secrets.ARIA_API_URL }}/health`
   - Verifica resposta 200
   - Falha o workflow se a API não responder em 30s
5. **Documentação** de como reativar/reconfigurar o UptimeRobot caso o monitor seja deletado.

## Scope

**IN:**
- Configurar UptimeRobot (via UI — não há API IaC)
- Adicionar GitHub Actions secret `ARIA_API_URL`
- Criar `.github/workflows/smoke-test.yml`
- Documentar configuração do UptimeRobot

**OUT:**
- Não configurar alertas por email (apenas Telegram)
- Não configurar Render Metrics (plano pago)
- Não monitorar endpoints além do `/health`

## Dependencies

- Story 11.1 concluída: Dockerfile funcionando
- Story 11.2 concluída: API rodando no Render com `/health`
- Story 9.x concluída: TELEGRAM_BOT_TOKEN e TELEGRAM_CHAT_ID configurados

## Business Value

O keep-alive elimina o problema de cold start do Render Free Tier. Sem ele, cada GitHub Actions cron teria 30s de latência adicional (risco de timeout em chains longas). O alerta Telegram fecha o loop de observabilidade para operação autônoma.

## Tasks / Subtasks

- [ ] **Task 1** — Criar `.github/workflows/smoke-test.yml` (AC: 4)
  ```yaml
  name: Smoke Test — Aria API

  on:
    workflow_dispatch:
    schedule:
      - cron: '0 9 * * 1'  # Segunda às 9h (após deploy semanal)

  jobs:
    smoke:
      runs-on: ubuntu-latest
      steps:
        - name: Health check
          run: |
            response=$(curl --silent --max-time 30 --write-out "%{http_code}" \
              --output /dev/null "${{ secrets.ARIA_API_URL }}/health")
            if [ "$response" != "200" ]; then
              echo "❌ Health check failed: HTTP $response"
              exit 1
            fi
            echo "✅ Aria API healthy: HTTP $response"
  ```

- [ ] **Task 2** — Configurar UptimeRobot (AC: 1, 2) — manual via UI

## Dev Notes

### UptimeRobot Telegram Integration
1. No bot do Aria (@aria_bot ou o bot configurado): adicionar `/start` no chat com o bot
2. No UptimeRobot: Alert Contacts → Add Alert Contact → Telegram
3. Usar o TELEGRAM_BOT_TOKEN e TELEGRAM_CHAT_ID das configs existentes

### Render Free Tier Cold Start
- Spin-down ocorre após 15 min de inatividade
- UptimeRobot pingando a cada 5 min mantém o serviço quente
- Ping de 5 min usa ~8640 requisições/mês (bem abaixo do limite free do UptimeRobot: 50 monitors)

### GitHub Actions Secret
```bash
gh secret set ARIA_API_URL --body "https://aria-api.onrender.com"
```

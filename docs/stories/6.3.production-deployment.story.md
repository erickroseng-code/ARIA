## Story

**As a** sistema,
**I want** estar deployado em produção em Railway (frontend) e VPS (backend),
**so that** usuários acessem aplicação 24/7 com uptime ≥99.5%.

## Acceptance Criteria

1. Frontend (Next.js) deployado em Railway com auto-deploy via GitHub
2. Backend (Fastify) deployado em VPS (DigitalOcean/Hetzner) com PM2
3. Environment variables: .env.production com secrets via Railway/GitHub Secrets
4. Database: PostgreSQL em produção (migração automática via Prisma)
5. Redis: Cache layer em produção (Railway Redis ou VPS self-hosted)
6. SSL/TLS: HTTPS forçado, certificados via Let's Encrypt (Certbot)
7. Health checks: `/health` endpoint respondendo 200 em <100ms
8. Uptime monitoring: StatusPage.io ou Uptime Robot (alertas)
9. Log aggregation: Winston logs em arquivo + rotação diária
10. Backup: Database backup diário (S3 ou backup service)
11. CI/CD: GitHub Actions com build, test, deploy em staging e prod
12. Rollback procedure: Documentado e testado

## Risks & Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|-----------|
| **Deployment fails** | Service outage | Low | Test CI/CD in staging first; have rollback ready |
| **Database migration fails** | Data loss or corruption | Low | Backup before migration; test in staging |
| **Secrets leaked** | Security breach | Low | Use GitHub Secrets; audit access logs |
| **High latency (>500ms)** | Poor user experience | Medium | CDN for static assets; database optimization |
| **Certificate renewal fails** | Service downtime (HTTPS) | Low | Setup auto-renewal; monitor expiry dates |
| **Disk space full** | Log files crash system | Low | Configure log rotation; monitor disk usage |

## Out of Scope

- ❌ **Kubernetes deployment** — VPS only for MVP
- ❌ **Multi-region deployment** — Single region only
- ❌ **CDN setup** — Not in MVP scope
- ❌ **Load balancer** — Single backend instance
- ❌ **Docker containers** — Direct VPS installation
